## 1. 서론

### 1.1 문서의 목적 및 범위

본 프로젝트 계획서는 **MiniDrive** 시스템 개발에 대한 전반적인 내용을 정리하고, 프로젝트를 진행하는 데 있어 각 세부사항의 지표로 삼는 것이 목적이다.

### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의

조직 내 파일을 공유하는 과정에서 발생하는 버전 관리의 어려움과 분산된 파일 문제를 해결하기 위한 클라우드 기반 파일 관리 시스템을 개발한다. 약 200명의 사용자가 웹 브라우저를 통해 파일을 업로드, 다운로드, 공유할 수 있고, 버전 기록 및 접근 권한 관리를 통해 데이터를 안전하게 보호한다.

#### 1.2.2 주요 기능 설명

| 기능 번호 | 설명 |
|:---:|:---|
| 01 | 다양한 형식의 파일을 업로드, 다운로드 가능 |
| 02 | 프로젝트별 폴더 생성 및 파일 정리, 삭제된 파일 복구 가능 |
| 03 | 사용자별로 보기, 수정, 댓글 권한 설정 및 외부 공유 링크 생성 |
| 04 | 파일명, 날짜, 업로더, 파일 유형 기반의 검색 및 필터링 |
| 05 | 파일 수정 시 이전 버전 자동 보관 및 복원 |
| 06 | 관리자 계정 생성/삭제, 로그인 인증, 데이터 암호화 전송 |

### 1.3 프로젝트 목표

조직 구성원 모두가 별도의 프로그램 설치 없이 웹 브라우저로 파일을 안전하게 저장하고 공유할 수 있는 시스템을 개발한다. 파일의 버전 관리와 권한 제어를 통해 데이터의 보안성과 팀 협업 효율성을 높이는 것이 목표이다. 

### 1.4 용어 정의

| 용어 | 설명 |
|:---:|:---|
| 버전 관리 | 파일의 수정 이력을 보관하여 이전 상태로 되돌릴 수 있도록 하는 기능 |
| 공유 링크 | 외부 사용자가 파일에 접근할 수 있도록 생성하는 URL (만료 기간 설정 가능) |
| 접근 권한 | 사용자별로 파일을 볼 수 있는지, 수정할 수 있는지를 나누는 설정 |
| 메타데이터 | 파일명, 업로드 날짜, 크기, 업로더 등 파일에 관한 부가 정보 |

## 2. 개발 계획

### 2.1 개발 절차 모형

#### 1) 개발 절차 모형

**점진적 모델**을 사용한다.

MiniDrive는 파일 저장, 공유, 검색, 버전 관리 등 기능이 명확하게 나뉜다. 핵심 기능(업로드/다운로드)을 먼저 구현하고 이후 기능을 하나씩 추가해가는 방식이 팀 규모와 일정에 적합하다고 판단했다.

#### 2) 분석·설계 방법론

**객체지향 방법론**을 사용한다. 파일, 사용자, 폴더, 권한 등을 독립적인 객체로 모델링해 기능 추가 및 수정이 용이하도록 설계한다.

### 2.2 WBS (Work Breakdown Structure)

```
MiniDrive 개발
├── 요구분석
│   ├── 유사 시스템 조사 (Google Drive 등)
│   ├── 요구사항 정의 및 분석
│   └── 핵심 기능 정의
├── 설계
│   ├── 시스템 구조 설계
│   └── DB 설계 및 화면 설계
├── 구현
│   ├── Increment 1: 파일 업로드/다운로드
│   ├── Increment 2: 폴더 구조 및 휴지통
│   ├── Increment 3: 권한 제어 및 공유 링크
│   ├── Increment 4: 검색 및 필터링
│   ├── Increment 5: 버전 관리 및 복구
│   └── Increment 6: 관리자 기능 및 보안
└── 테스트
    ├── 기능 테스트
```

## 4. 품질 관리

### 4.1 팀 미팅 계획

**(1) 주간 팀 회의**
- 미팅 시간: 매주 금요일 오후 6시
- 토의 내용: 진행 상황 공유, 다음 주 할 일 정리, 이슈 공유
- 참석자: 팀원 전체, 팀장

**(2) Increment 검토 회의**
- 미팅 시간: 각 Increment 완료 시점
- 토의 내용: 구현된 기능 확인 및 다음 Increment 계획 수립
- 참석자: 팀원 전체, 팀장

**(3) 비정기 회의**
- 문제가 발생하거나 논의가 필요한 경우 수시로 진행
- 팀장 또는 지도 교수에 의해 요청될 수 있다
- 참석자: 팀원, 팀장, (지도 교수)

### 4.2 변경 사항 관리 방법

- 기능 변경은 팀원 전체의 합의가 필요하며 다수결로 결정한다.
- 변경 사항이 발생하면 변경 전·후 내용을 반드시 기록으로 남긴다.
- 관련 산출물은 팀원 전체가 소지한다.
- 의견 충돌로 결론이 나지 않을 경우 팀장이 최종 결정권을 갖는다.
- 변경 사항이 결정되면 지도 교수님께 보고하고 피드백을 반영한다.

### 4.3 산출물 관리 방법

**(1) 산출물 명명(Naming) 방법**
- 소프트웨어: `버전_날짜` (예: `v1_250601.jar`)
- 문서: `문서명_YYMMDD_Doc-버전` (예: `프로젝트관리계획서_250601_Doc-001.md`)
- 그 외 파일: 목적이 드러나는 이름

**(2) 버전 관리 방법**
- 소프트웨어: 파일명 앞에 버전 기재
- 문서: 파일명에 날짜 기재 + 문서 내 수정 이력 수정 (팀장)
- 그 외 파일: 파일명 앞에 날짜 기재

**(3) 저장 방법**
- 소프트웨어: GitHub
- 문서 및 기타 파일: GitHub 및 팀원 공유 클라우드
